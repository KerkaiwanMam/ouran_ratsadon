# Fiscal Intelligence: Macro Overview, Recipient Tracking & Company Verification

> ส่วนขยายจาก CLAUDE.md / roadmap.md — เพิ่มมุมมอง "บริบทระดับประเทศ" (รายได้/รายจ่าย/ดุล/หนี้สาธารณะ) และ "ตรวจสอบผู้รับงบ/บริษัท" เข้ากับ Civic Layer พร้อมยก UX ของกราฟแนวโน้มสไตล์ TradingView (อ้างอิง interaction ไม่ใช่ embed widget)
>
> **ข้อจำกัดสำคัญที่ต้องรับรู้ก่อนเริ่ม**: ข้อมูลปัจจุบันใน `data/budget-XXXX.json` เป็น "การจัดสรรงบรายจ่าย" (expenditure allocation) เท่านั้น — ไม่มีข้อมูลรายได้รัฐ/หนี้สาธารณะ/ผู้รับเหมารายโครงการ ฟีเจอร์นี้ต้องการ **แหล่งข้อมูลใหม่** (ดูหัวข้อ "แหล่งข้อมูล" แต่ละส่วน) และควรเริ่มจากขอบเขตที่มีข้อมูลพร้อมจริงก่อน ไม่ใช่ design ทุกอย่างพร้อมกันแล้วติดที่ data sourcing

## ขอบเขต (ตามที่ยืนยัน)

ครอบคลุมทั้งสองระดับ:
1. **ระดับมหภาค (Macro Fiscal)** — งบประมาณแผ่นดินทั้งระบบ: รายได้-รายจ่าย-ดุล-หนี้สาธารณะ ต่อปี
2. **ระดับโครงการ/หน่วยงาน** — ขยายข้อมูลที่มีอยู่ใน `/project/[id]` และ `/explore` ให้เห็น "ใครรับงบไปทำอะไร"

และครอบคลุมการตรวจสอบบริษัททั้งสองแบบ:
1. **ทะเบียนผู้รับงบ + red flag** (แบบ USASpending "Recipient") — ประวัติการรับงาน, ความเสี่ยง, การกระจุกตัว
2. **วิเคราะห์งบการเงินบริษัท** (แบบ `liberator-finance-suite`) — สำหรับบริษัทมหาชนที่รับงานรัฐ ดูสุขภาพการเงินประกอบการตัดสินใจ/ความน่าเชื่อถือ

## ส่วนที่ 1 — Macro Fiscal Overview (หน้าใหม่ใน Civic Layer)

### UI/Behavior
- หน้าใหม่ เช่น `/fiscal-overview` — แสดง 4 ตัวเลขหลักเป็น stat strip: รายได้รัฐ, รายจ่ายรัฐ, ดุลงบประมาณ (เกินดุล/ขาดดุล), หนี้สาธารณะ (+ สัดส่วนต่อ GDP ถ้ามีข้อมูล)
- กราฟแนวโน้มหลายปี (interactive trend chart) แสดงเส้นรายได้/รายจ่าย/หนี้สาธารณะซ้อนกัน พร้อม:
  - **Zoom/pan ตามช่วงปี** และปุ่มเลือกช่วงเวลาด่วน (5Y / 10Y / All) — อ้างอิง UX ของ TradingView range selector
  - **Toggle overlay** เปิด/ปิดแต่ละเส้นข้อมูล (รายได้/รายจ่าย/หนี้) เพื่อเปรียบเทียบเฉพาะคู่ที่สนใจ
  - **Crosshair tooltip** เมื่อ hover แสดงตัวเลขของปีนั้นๆ ครบทุกตัวชี้วัด ไม่ต้องสลับกราฟไปมา
- เชื่อมกับ `/explore`: ใส่ context bar ที่หน้า explorer บอกว่า "งบรายจ่ายปีนี้ = X% ของรายได้รัฐ" หรือ "ปีนี้ขาดดุล Y บาท" — ทำให้ตัวเลขการจัดสรรงบที่ผู้ใช้กำลังดูมี "บริบทระดับประเทศ" ไม่ใช่ตัวเลขลอยๆ

### Data model

```prisma
model FiscalYearSummary {
  id              String  @id @default(cuid())
  fiscalYear      String  @unique // "2568"
  totalRevenue    Float
  totalExpenditure Float
  balance         Float   // revenue - expenditure (ติดลบ = ขาดดุล)
  publicDebt      Float
  debtToGdpPct    Float?
  source          String  // อ้างอิงแหล่งข้อมูล (สำนักงบประมาณ / สบน. / คลัง)
  publishedAt     DateTime?
}
```

> เก็บแยกจาก civic budget tree (`data/budget-XXXX.json` → ministries/departments/projects ใน in-memory cache) เพราะเป็นข้อมูลคนละ "หน่วย" (ภาพรวมประเทศ vs การจัดสรรรายกระทรวง) — รวมกันใน UI ผ่าน API join ไม่ใช่ schema เดียวกัน

### แหล่งข้อมูล (ต้องจัดหาก่อนเริ่ม build)
- รายได้รัฐ + ดุลงบประมาณ: รายงานฐานะการคลัง สำนักงานเศรษฐกิจการคลัง (สศค.) / กรมบัญชีกลาง
- หนี้สาธารณะ: สำนักงานบริหารหนี้สาธารณะ (สบน.) — มีรายงานสถานะหนี้สาธารณะรายเดือน/รายปีแบบเปิด
- เช่นเดียวกับ Civic data เดิม: pre-process เป็น JSON แบบ offline ใส่ไว้ใน `data/fiscal-summary.json` แล้ว seed ลง `FiscalYearSummary` — **ไม่ใช่ live API call** (สอดคล้องกับแนวทาง pre-processed civic data ที่ใช้อยู่)

## ส่วนที่ 2 — ผู้รับงบ/หน่วยงานระดับโครงการ

### UI/Behavior
- ขยาย `/project/[id]`: เพิ่ม section "ผู้ดำเนินการ/ผู้รับเหมา" แสดงชื่อบริษัท/หน่วยงานที่ได้รับงบ (ถ้ามีข้อมูล) พร้อมลิงก์ไปหน้าโปรไฟล์ผู้รับงบ (ส่วนที่ 3)
- หน้าใหม่ `/recipient/[id]` (คล้าย USASpending recipient profile): ประวัติการรับงบจากภาครัฐ (กี่โครงการ, รวมกี่บาท, จากกระทรวงไหนบ้าง), แนวโน้มย้อนหลัง

### ข้อจำกัดข้อมูล (สำคัญ — ต้อง flag ไว้ตั้งแต่ planning)
ข้อมูลผู้รับเหมา/คู่สัญญาระดับโครงการของไทยมักอยู่ใน **ระบบจัดซื้อจัดจ้างภาครัฐ (e-GP ของกรมบัญชีกลาง)** ซึ่งเป็นคนละชุดข้อมูลกับงบประมาณรายจ่าย (`budget-XXXX.json` ปัจจุบัน) และการเชื่อมสองชุดเข้าด้วยกัน (โครงการ ↔ สัญญา ↔ คู่สัญญา) ไม่ตรงไปตรงมา — **แนะนำให้กำหนดเป็น stretch goal ของ Phase 2** และเริ่มจาก subset เล็กๆ (เช่น โครงการลงทุนขนาดใหญ่ที่มีข้อมูลคู่สัญญาเปิดเผยชัดเจน) แทนการพยายาม cover ทุกโครงการตั้งแต่แรก

## ส่วนที่ 3 — ตรวจสอบบริษัท (สองแบบตามที่ยืนยัน)

### 3a. ทะเบียนผู้รับงบ + Red flag (ต่อยอดจากระบบ red flag ที่มีอยู่)
- เพิ่ม red-flag rule ระดับ "ผู้รับงบ" (ไม่ใช่ระดับโครงการเดี่ยว) เช่น:
  - **การกระจุกตัว**: บริษัทเดียวรับงานจากกระทรวง/กรมเดียวกันซ้ำๆ ในสัดส่วนสูงผิดปกติ
  - **อายุ/ขนาดไม่สอดคล้องกับมูลค่างาน**: บริษัทเพิ่งจดทะเบียน/ทุนจดทะเบียนต่ำ แต่ได้งานมูลค่าสูง (ต้องเชื่อมกับข้อมูลกรมพัฒนาธุรกิจการค้า — ก็เป็นอีกแหล่งข้อมูลที่ต้องจัดหา)
  - **ผู้รับงบซ้ำที่มีโครงการติดธงแดงสะสม**: ลิงก์กับ `RedFlag` ที่มีอยู่แล้วในระบบ ไม่ต้องสร้างใหม่ทั้งหมด — แค่ aggregate ขึ้นมาที่ระดับผู้รับงบ
- แสดงผลเป็น "คะแนนความเสี่ยง" + คำอธิบายเป็นภาษาไทยแบบเดียวกับ `RedFlag.description` ปัจจุบัน — คงหลัก "อธิบายเหตุผลตรงไปตรงมา ไม่ใช่ black-box score"

### 3b. วิเคราะห์งบการเงินบริษัทมหาชน (ต่อยอดแนวคิดจาก `liberator-finance-suite`)
- สำหรับบริษัทที่ (ก) จดทะเบียนในตลาดหลักทรัพย์ และ (ข) ปรากฏเป็นผู้รับงบในข้อมูล Civic — แสดงข้อมูลการเงินพื้นฐานประกอบ: งบกำไรขาดทุนย่อ, อัตราส่วนสภาพคล่อง/หนี้สิน, แนวโน้มรายได้
- จุดประสงค์: ช่วยตอบคำถาม "บริษัทนี้มีศักยภาพการเงินเพียงพอจะรับงานมูลค่าขนาดนี้ไหม" — เป็นข้อมูลประกอบการตัดสินใจของผู้ใช้ ไม่ใช่การให้คำแนะนำการลงทุน (สอดคล้องกับกฎ "ไม่ให้คำแนะนำการลงทุนแบบ personalized")
- แหล่งข้อมูล: งบการเงินที่เปิดเผยต่อ ก.ล.ต./ตลาดหลักทรัพย์ฯ (SET) — เป็นข้อมูลสาธารณะ แต่ปริมาณงาน parse/normalize สูง ควรจำกัดเฉพาะบริษัทที่ปรากฏในข้อมูล Civic จริงๆ (lazy-load ตามที่ต้องใช้ ไม่ preprocess ทั้งตลาด)
- **คำเตือนเรื่อง scope**: นี่คือจุดที่เสี่ยงต่อการ "บวมเป็นโปรเจกต์ที่สอง" มากที่สุด — แนะนำเริ่มจากแสดงข้อมูลพื้นฐาน (ไม่ใช่ full stock-analysis suite) แล้วประเมินว่าคุ้มที่จะขยายไหมจาก feedback จริง

## TradingView-inspired chart UX — ข้อเสนอด้านเทคนิค

> ตามที่ยืนยัน: ใช้ "อ้างอิง UX" ไม่ใช่ embed widget จริง — เพื่อให้คุมธีม/ภาษา/responsive ได้เอง และไม่ผูกกับ TradingView's branding/licensing

- **Recharts ปัจจุบันไม่รองรับ zoom/pan แบบ native ดีนัก** — ตัวเลือกที่เหมาะ:
  - **`lightweight-charts`** (โอเพนซอร์สจาก TradingView เอง, MIT license) — มี zoom/pan/crosshair/range-selector ในตัว, น้ำหนักเบา, เหมาะสำหรับ time-series แบบที่ใช้ในส่วนนี้พอดี
  - หรือสร้างเอง บน D3 (ที่มีอยู่แล้วในสแตก) ถ้าต้องการคุม styling ละเอียดกว่า — ใช้เวลามากกว่า
- แนะนำ **`lightweight-charts`** สำหรับ Macro Fiscal trend chart และ project 5-year history (ยกระดับจาก bar chart ธรรมดาที่มีอยู่) — ลด custom code, ได้ UX ระดับ TradingView ตรงตามที่ขอ โดยไม่ต้อง build interaction เองทั้งหมด

## ผูกกับ Roadmap

แนะนำวาง **เป็น Phase 2 ทั้งชุด** (ไม่ใช่ Phase 0/1) เพราะ:
- ต้องจัดหา/ประมวลผลชุดข้อมูลใหม่ทั้งหมด (รายได้รัฐ, หนี้สาธารณะ, คู่สัญญา, งบการเงินบริษัท) ซึ่งเป็นงาน data-sourcing ที่ใหญ่กว่าโค้ด
- Phase 0 ต้องโฟกัส demo Civic Layer ที่มีอยู่ให้เสร็จก่อนตามที่ `CLAUDE.md` กำหนดไว้ ("do not start Business Layer until Civic 0a is demo-ready" — หลักการเดียวกันใช้ได้กับฟีเจอร์ใหญ่ใหม่ๆ ด้วย คือ อย่าเริ่มจนของเดิมเสร็จ)
- ลำดับที่แนะนำภายใน Phase 2: เริ่มจาก **ส่วนที่ 1 (Macro Fiscal Overview)** ก่อน เพราะมีแหล่งข้อมูลชัดเจนที่สุดและไม่ต้อง join ข้ามชุดข้อมูล จากนั้นค่อย 3a (red flag ระดับผู้รับงบ — ต่อยอดของเดิม) แล้วค่อยพิจารณา ส่วนที่ 2 และ 3b เป็น stretch ถ้ามีเวลา/ข้อมูลพร้อม
