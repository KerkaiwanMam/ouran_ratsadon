🔵 Phase A — Infra & Security Foundation Blocker ของทุก phase · เริ่มวันแรก
A0
Pre-flight: Env & Secrets inventory DevOps

รวบรวม env vars ทั้งหมดลง .env.example: DATABASE_URL (Neon), UPSTASH_REDIS_REST_URL/TOKEN, R2_ACCOUNT_ID / ACCESS_KEY_ID / SECRET_ACCESS_KEY / BUCKET, STRIPE_SECRET_KEY / STRIPE_WEBHOOK_SECRET, CRON_SECRET — ตั้งค่าใน Vercel + Railway ทั้ง preview/production

✓ .env.example ครบ ทุก env ตั้งใน Vercel/Railway แล้ว
A1
Neon DB production hardening DevOpsBE

แยก Neon branch main (prod) / dev (staging) · เปิด connection pooling (pgbouncer) แล้วใช้ pooled connection string ใน Prisma · รัน prisma migrate deploy · ตรวจ index ตาม docs/database-schema.md · ตั้ง backup/PITR

✓ migrate ผ่านบน 2 branch · query /api/civic/search เร็วปกติ
A2
Distributed Rate Limit ด้วย Upstash Redis depends: A0 BE

ติดตั้ง @upstash/ratelimit + @upstash/redis · แทนที่ in-process Map<string, RateBucket> ใน apps/web/middleware.ts ด้วย sliding window · คง tier เดิม: upload 5/min, auth 10/min, api 60/min · เปิด ephemeralCache ลดจำนวน Redis call · fail-open: ถ้า Redis ล่ม fallback ไป in-memory Map เดิม ห้าม block ผู้ใช้ทั้งระบบ

✓ rate limit นับข้าม region/instance ได้ · ยังเป็น Step 1 ก่อน JWT decode
A3
Middleware regression test depends: A2 BE

เขียน test ยืนยันลำดับ: ① rate limit → ② JWT decode → ③ route matching · ยิง 6 req/min ใส่ /api/files/upload ต้องได้ 429 ตัวที่ 6 · ตรวจ header X-RateLimit-Remaining

✓ test ผ่าน · 429 + Thai error message ถูกต้อง
🟢 Phase B — Storage: R2 Presigned URL + Malware Scan depends: Phase A
B1
R2 bucket + CORS + lifecycle DevOps

สร้าง bucket แยก prod/staging · ตั้ง CORS ให้รับ PUT จาก origin ของแอปเท่านั้น · lifecycle rule ลบ object ที่ key prefix quarantine/ อายุ > 1 วันอัตโนมัติ

✓ PUT จาก origin อื่นถูก CORS ปฏิเสธ
B2
API ออกตั๋ว: POST /api/files/presign depends: B1 BE

ตรวจ auth + tier quota (Free 3 ไฟล์/เดือน) → validate นามสกุลตาม whitelist + declared size ≤ 20MB → ออก presigned PUT URL อายุสั้น (~10 นาที) ผูก Content-Length เงื่อนไขบน R2 → สร้างแถว FileUpload status=UPLOADING · ใช้ rate limit tier upload 5/min เดิม

✓ ไฟล์ >20MB หรือนามสกุลผิดถูกปัดตั้งแต่ presign ไม่แตะ memory
B3
Client direct-upload + confirm depends: B2 FEBE

FE อัปโหลดตรงเข้า R2 พร้อม progress bar → เรียก POST /api/files/complete → status=PROCESSING → enqueue งานสแกน · แสดงสถานะ "กำลังตรวจสอบไฟล์…" เป็นภาษาไทย

✓ อัปโหลด 19MB สำเร็จ เห็น progress จริง
B4
FastAPI scan worker — สแกนแล้วลบทันที depends: B3 BE

Parser service ดึง object จาก R2 → ตรวจตามลำดับ: ① magic bytes ตรงนามสกุลจริง ② containsMacros() (VBA/macro) ③ zip-bomb / nested archive depth ④ ClamAV scan (ผ่าน clamd container ข้าง FastAPI) · fail ข้อใดข้อหนึ่ง → ลบ object ออกจาก R2 ทันที + status=ERROR + audit log · ผ่านครบ → ส่งต่อ parse pipeline เดิม (sanitizeStringField ยังทำงานก่อน DB insert)

✓ ไฟล์ EICAR ถูกลบจาก R2 ภายใน scan cycle เดียว + user เห็น error ไทย
B5
Orphan cleaner: ปิด TODO stub depends: B4 BE

เติม logic ลบ R2 object จริงใน lib/orphan-cleaner.ts (stub ที่ค้างจาก Fix #4) · ตั้ง Vercel Cron */30 * * * * ยิง GET /api/internal/cleanup-orphans พร้อม CRON_SECRET

✓ orphan UPLOADING >30 นาที ถูกลบทั้ง DB row และ R2 object
🟣 Phase C — Stripe Webhook & Tier Management depends: A1 เท่านั้น — ทำขนานกับ B ได้
C1
Webhook endpoint + signature verify BE

POST /api/webhooks/stripe อ่าน raw body (ห้ามให้ Next.js parse JSON ก่อน) → stripe.webhooks.constructEvent() ด้วย STRIPE_WEBHOOK_SECRET · ตอบ 200 เร็วที่สุด แล้วประมวลผลต่อ · ใส่ route นี้ใน middleware matcher แบบไม่ติด auth (Stripe ไม่มี JWT) แต่ยังอยู่ใต้ rate limit tier api

✓ payload ปลอม signature ผิด → 400 ทุกครั้ง
C2
แกะ 3 event หลัก → จัดการ tier depends: C1 BE

① invoice.paid / checkout.session.completed → activate/renew Pro (฿299) หรือ Team (฿799) ตาม price ID · ② customer.subscription.updated → sync เปลี่ยนแผน/upgrade/downgrade + สถานะ past_due → grace period · ③ customer.subscription.deleted → downgrade เป็น FREE ทันทีเมื่อสิ้นรอบบิล · อย่าแตะ isManuallyGranted — ให้ webhook เขียนเฉพาะ subscription ที่มาจาก Stripe

✓ ทั้ง 3 event เปลี่ยน tier ใน DB ถูกต้อง · manual grant ไม่ถูก override
C3
Idempotency + retry safety depends: C2 BE

ตาราง StripeEventLog เก็บ event.id แบบ unique — event ซ้ำ (Stripe retry) ต้อง no-op · ครอบ handler ด้วย prisma.$transaction ป้องกัน partial update

✓ ยิง event เดิมซ้ำ 3 ครั้ง ผล DB เท่ากับ 1 ครั้ง
C4
ทดสอบครบวงจรด้วย Stripe CLI depends: C3 BEDevOps

stripe listen --forward-to + stripe trigger ทั้ง 3 event · ทดสอบ flow จริง: checkout → trial 14 วัน → paid → cancel · ลงทะเบียน webhook endpoint บน Stripe Dashboard (test + live แยกกัน)

✓ subscribe→cancel ครบ loop บน test mode โดยไม่แก้ DB มือ
🟠 Phase D — Frontend Redesign: 2 โหมด 2 บุคลิก เริ่มขนานได้ตั้งแต่ A เสร็จ · D4 รอ B
D1
Design tokens 2 theme ใน Tailwind FE

อ่าน docs/design-system.md ก่อน · กำหนด CSS variables แยก route group: Civic "ผังเมือง" — โทน navy/slate + amber accent, grid line แบบ blueprint, typography ราชการอ่านง่าย · Business "โมเดิร์นเอกชน" — โทนขาวสะอาด + emerald/teal accent, การ์ดเงาเบา แบบ SaaS · ห้าม inline style, ใช้ Tailwind + shadcn/ui เท่านั้น

✓ theme สลับตาม layout ของ (civic) / (business) อัตโนมัติ
D2
Layout shell + navigation refresh depends: D1 FE

ปรับ shell ของ (civic): header แบบ public portal, breadcrumb, fiscal context bar · shell ของ (business): sidebar dashboard, workspace switcher · ตรวจ responsive + dark mode ทั้ง 2 โหมด

✓ ทุกหน้าหลักใช้ shell ใหม่ ไม่มีหน้า "ตกค้าง" theme เก่า
D3
Component refresh รายหน้า depends: D2 FE

Civic: /explore treemap + drill-down, /search, /project/[id], /fiscal-overview, /compare · Business: dashboard, /analytics (4 tier), settings · คง OG image + share ต่อหน้า · UI text ไทยทั้งหมด

✓ visual QA ผ่านทุกหน้า ทั้ง light/dark · Lighthouse a11y ≥ 90
D4
Upload UX ใหม่ (presigned flow) depends: B3, D2 FE

หน้าอัปโหลดรองรับ 3 สถานะใหม่: กำลังอัปโหลด (progress %) → กำลังสแกนไฟล์ → ผลลัพธ์ (ผ่าน/ติดมัลแวร์/เกิน 20MB) · error message ไทยเข้าใจง่าย · แสดง quota คงเหลือของ Free tier

✓ ทุก error path มีข้อความไทย ไม่มี raw error หลุดถึง user
D5
Pricing + Billing UI sync กับ Stripe depends: C2, D2 FE

หน้า pricing แสดง Free / Pro ฿299 / Team ฿799 · billing settings แสดงสถานะ subscription จริงจาก webhook (active, trial, past_due, canceled) + ปุ่ม manage ผ่าน Stripe Customer Portal

✓ เปลี่ยนแผนบน Stripe test แล้ว UI อัปเดตตรงตามจริง
🟩 Phase E — QA → Staging → Production Launch depends: B, C, D ครบ
E1
Integration test ตาม DoD BEFE

ไล่ checklist ใน tab "MVP Definition of Done" ทุกข้อบน staging (Vercel preview + Neon dev branch + R2 staging bucket + Stripe test mode)

✓ DoD ผ่าน 100% บน staging
E2
Production cutover depends: E1 DevOps

สลับ env เป็น production ทั้งชุด → prisma migrate deploy บน Neon main → ลงทะเบียน Stripe live webhook → ตั้ง Vercel Cron orphan cleanup → smoke test: civic explore, สมัครสมาชิก, อัปโหลด, checkout จริง 1 รายการแล้ว refund

✓ smoke test ผ่านบน production จริง
E3
Post-launch monitoring (48 ชม.แรก) depends: E2 DevOps

เฝ้าดู: Upstash analytics (อัตรา 429), Stripe webhook delivery success rate, Neon connection count, error log ของ FastAPI scan worker · เตรียม rollback plan: Vercel instant rollback + Neon PITR

✓ ไม่มี incident P1 ใน 48 ชม. · webhook success ≥ 99%
